# Demo Script Template - Template Skrip Demo untuk Hackathon

## Overview

Demo adalah momen paling penting dalam hackathon presentation. Judges ingin melihat produk Anda bekerja secara real-time, bukan hanya screenshot atau video. Template ini menyediakan struktur lengkap untuk membuat demo script yang smooth dan impressive.

## 1. Demo Flow Sequence

### 1.1 Standard Demo Flow (5 menit)

```
┌─────────────────────────────────────────────────────────────────┐
│                    5-MINUTE DEMO FLOW                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│ 0:00-0:30 │ OPENING (30 detik)                               │
│            │ • Hook audience dengan problem statement          │
│            │ • Briefly introduce your solution                  │
│            │                                                      │
│ 0:30-1:00 │ SETUP (30 detik)                                 │
│            │ • Show empty state / initial screen               │
│            │ • Connect wallet (show it's easy)                 │
│            │                                                      │
│ 1:00-2:30 │ CORE FEATURE DEMO #1 (1.5 menit)                  │
│            │ • Primary user action                             │
│            │ • Show transaction in progress                    │
│            │ • Confirm success with visual feedback             │
│            │                                                      │
│ 2:30-3:30 │ CORE FEATURE DEMO #2 (1 menit)                   │
│            │ • Secondary feature showing depth                 │
│            │ • Quick and impactful                              │
│            │                                                      │
│ 3:30-4:30 │ EDGE CASE/ADVANCED (1 menit)                       │
│            │ • Show you thought about edge cases               │
│            │ • Advanced feature atau integration               │
│            │                                                      │
│ 4:30-5:00 │ CLOSING (30 detik)                                │
│            │ • Recap value proposition                         │
│            │ • Call to action                                  │
│            │ • Thank audience                                 │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 Demo Flow by Project Type

**DeFi Protocol Demo Flow:**

```
1. Show current DeFi problem (high fees, slow, complex)
2. Connect wallet (seamless)
3. Deposit/Stake tokens (show gas estimate)
4. Show rewards/earnings (real-time update)
5. Withdraw (show it works)
6. Close with "Imagine doing this with 80% less fees"
```

**NFT Marketplace Demo Flow:**

```
1. Show pain of current NFT buying (gas wars, complexity)
2. Browse marketplace (fast, beautiful UI)
3. Mint NFT (lazy minting - no upfront gas)
4. List for sale (set price, fees shown)
5. Show listing (appears instantly)
6. Close with "NFT for everyone, not just whales"
```

**DAO Tooling Demo Flow:**

```
1. Show DAO pain (hard to track proposals, low participation)
2. Dashboard overview (clear, organized)
3. Create proposal (simple form)
4. Vote from dashboard (one-click)
5. Show results (real-time tally)
6. Close with "Governance for humans, not just power users"
```

### 2.2 Backup Timing Buffer

Selalu punya buffer untuk:

- Transaction confirmation delays: +30 detik
- Technical issues: +1 menit
- Questions from judges during demo: +1 menit

Target: Demo yang butuh 5 menit, selesai dalam 4 menit max untuk give buffer.

```

│                                                                  │
│ SCENARIO 1: Wallet Connection Fails                            │
│ ├─ Backup: Pre-connected wallet screenshot                     │
```

### 3.2 Backup Preparation Checklist

```
□ Pre-recorded video of full demo (MP4)
□ Screenshots of each step (PNG/JPG)
□ Multiple browser tabs with different states
□ Wallet already connected in backup tab
□ Contract addresses documented
□ Block explorer links ready
□ PDF/printout of demo steps (just in case)
```

## 4. Live Coding Moments

### 4.1 When to Include Live Coding

Live coding bisa impressive jika:

- Anda expert di area tersebut
- Coding adalah bagian dari value proposition
- Ada time remaining untuk show depth

### 4.2 Live Coding Script

**If including live coding (max 30 seconds):**

```
"At the core of our protocol is this smart contract function
that calculates rewards. Let me show you..."

[Open contract in browser/remix]
"Here's the staking function - you can see we use
a compounding algorithm with a rate of..."

[Highlight key line]
"This is what makes our protocol 40% more efficient
than existing solutions."

[Close quickly]
"And this all happens automatically on-chain."
```

### 4.3 Live Coding DON'Ts

- DON'T write new code during demo
- DON'T debug errors live
- DON'T explain every line
- DON'T exceed 30 seconds
- DON'T use live coding unless you're confident

## 5. Screen Sharing Checklist

### 5.1 Pre-Demo Setup

```
□ Browser cleaned (close unnecessary tabs)
□ Screen resolution tested
□ Font sizes readable (zoom in if needed)
□ Notifications disabled
□ Background apps closed
□ Desktop cleaned (no sensitive info visible)
□ Demo tab is first tab
□ Second monitor off atau showing backup
```

### 5.2 During Demo Visibility Tips

1. **Zoom appropriately** - Judges harus bisa baca text
2. **Highlight key actions** - Move mouse to important elements
3. **Pause for confirmation** - Wait for transactions
4. **Talk through actions** - Explain what showing
5. **Have backup** - Always have escape route

### 5.3 Browser Tab Organization

```
TAB 1: Main dApp (ready to connect wallet)
TAB 2: Block explorer (transaction details)
TAB 3: Backup dApp state (already connected)
TAB 4: Pre-recorded video (as last resort)
```

## 7. Practice & Rehearsal

### 7.1 Practice Schedule

```
PRACTICE SESSIONS:
□ Day before: Read through script (1x)
□ Day before: Practice with timer (2x)
□ Day of morning: Full rehearsal (2x)
□ Before presentation: Quick review (1x)
```

### 7.2 Practice Tips

1. **Practice out loud** - Hearing yourself helps
2. **Time yourself** - Know exactly how long each part takes
3. **Record yourself** - Review dan improve
4. **Practice transitions** - Between slides dan demo
5. **End on time** - Always better to have extra time

---

## Summary

Demo yang sukses membutuhkan:

1. **Script yang jelas** - Know exactly apa akan ditunjukkan dan kapan
2. **Timing yang tepat** - Stay within limits, have buffer
3. **Backup plans** - Always have escape routes
4. **Practice yang cukup** - Rehearse until comfortable
5. **Confidence** - Judges appreciate presenters who are confident

Tips terpenting:

- Never do live coding unless you're 100% confident
- Always have video backup ready
- Stay calm jika ada issues - judges understand
- Focus on making it look easy, even if it was hard to build
- Remember: They want you to succeed!
